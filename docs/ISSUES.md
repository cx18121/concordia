**#C1 — Governance must expose vote-readback ARRAY views** · OPEN (blocks resolve) · owner: ____ (workstream A). **Re-confirmed 6/13 against the LANDED `Governance.sol` (not just the interface).**
The contract *stores* the data — `address[] public voters` and `mapping(address => Alloc[]) public allocOf` — but the Solidity auto-getters are **index-based** (`voters(i)`, `allocOf(addr, i)`) with **no length and no whole-array return**, so the keeper can't enumerate them safely. (Scraping by index-looping until an OOB revert was considered and rejected: a transient RPC error mid-loop would silently truncate the voter set and mis-score people — worse than failing loud.) Fix is two trivial views; **recommended** (keeps the keeper's existing ABI unchanged — `getVoters()` + `allocOf(address)→Alloc[]`):
```solidity
mapping(address => Alloc[]) internal allocOf;            // drop `public` (the array view replaces the index getter)
function getVoters() external view returns (address[] memory) { return voters; }
function allocOf(address m) external view returns (Alloc[] memory) { return allocOf[m]; }
```
(If A would rather keep `allocOf` public, add `allocationsOf(address)→Alloc[]` instead and the keeper renames that one ABI entry — a one-line change in `keeper/src/chain/abi.ts` + `viemAdapter.ts`.) Until then the keeper's `readResolveInputs()` reverts **loudly** (never silently wrong). **Everything else the keeper calls already matches the landed contracts**: `resolveCycle(address[],int256[],uint256[])`, `setPrices(bytes32[],uint256[],uint256)`, `SyntheticExecutor.repeg(bytes32,uint256)`, `accuracyOf`, `openCycle`, `lockCycle` — exact signature matches. (Re-peg needs no `Vault.heldAssets()` — the keeper re-pegs a configured pool list.)

**#C2 — CRE write path needs an `onReport` receiver** · OPEN (CRE-only) · owner: ____ (workstream A). **Re-confirmed 6/13 against the LANDED contracts:** `PriceOracle` and `Governance` both gate writes on a single immutable `keeper` EOA (`onlyKeeper`), and **neither has an `onReport`** — so the **heartbeat (direct EOA calls) already matches the deployed contracts and is the integration-ready driver**; the CRE workflow's report path does not yet have a target.

## Resolved

*(answers folded in from research 6/12; design-changing ones also reflected in DESIGN.md / CONTRACTS.md)*

- **#14 — Position cap is a HARD limit via water-fill; baskets may hold cash (6/13).** Review caught that the spec'd "cap then renormalize" doesn't actually cap — renormalizing capped weights pushes them back over the cap (our own 2-asset case → 50% each under a 30% cap). Switched `selectBasket` to water-fill: pin over-cap names at the cap, redistribute to the rest, iterate. **Consequence:** if too few names are voted to fill 1e4 under the cap, the basket sums to < 1e4 and the rest stays in cash. **Keeper (C) + frontend (D): do not assume baskets sum to 100%.** Reflected in CONTRACTS §4 + DESIGN §3.
- **#15 — Reward pool gated on absolute dollar gain, not just relative alpha (6/13).** Review flagged that a positive benchmark-relative excess in a down market (market −10%, fund −5%) would fund a bonus out of shrinking principal. `settle` now also requires `NAVnow > navAtLock` and caps the rewardable gain at the real dollar gain; the relative high-water mark is *not* advanced on a down-money cycle (it pays later when the fund is up in absolute terms too). Reflected in DESIGN §3.

- **#1 — Price API: Yahoo Finance v8 chart API** (verified, returns data for all 19 tickers + S&P). No auth, no key — just a browser `User-Agent` header. **Live:** `GET query1.finance.yahoo.com/v8/finance/chart/{symbol}` → `.chart.result[0].meta.regularMarketPrice` (one call/ticker; v7 batch is locked). **Historical fixture:** same endpoint `?period1=1704067200&period2=1711929600&interval=1wk` → 13 weekly closes for early 2024; run once per ticker at kickoff, commit as JSON. **S&P 500 symbol = `%5EGSPC`** (URL-encoded `^GSPC`) — works for both. Rejected: Stooq (JS-challenge blocks server curl), xStocks API (metadata only, no prices, missing 17/18 tickers). Caveat: unofficial endpoint — add retries.
- **#2 — Use mock USDC.** Need six figures to seed 8 pools deep + fund 6 agents; Circle's faucet drips a few dollars. Mock USDC's public `mint()` doubles as the "get demo USDC" button. One constructor arg swaps to real USDC for production.
- **#3 — World ID verification: use the REST backend path.** A v3 `WorldIDRouter` *does* exist on Base Sepolia (`0x42FF98C4E85212a5D31358ACbFe76a621b50fC02`) but on-chain v3 is **Orb-only** (no good for ad-hoc judges) and v4's on-chain verifier is World Chain only. So verify via `POST developer.world.org/api/v4/verify/{rp_id}` (chain-agnostic, accepts non-Orb credentials) in a Next.js API route; the backend then marks the wallet verified in the Vault. This is explicitly allowed by World's prize rules. IDKit default level is `orb` — **must override** to a lower level (see #10).
- **#4 — AgentKit ↔ Dynamic server wallet: compatible, YES.** AgentKit registration is gated by a **human's World ID proof**, not an agent-wallet signature — the agent address is just data, so any address type registers. A hosted relay pays the registration gas (agent needs no funds). Registration happens on **World Chain (`eip155:480`)**, not Base. Request-time x402 signing needs a standard EIP-191 signature, which Dynamic MPC server wallets produce — works without special handling. (Note: AgentKit is v0.2.0 beta.)
- **#6 — Pool liquidity: seed ~100k USDC-equivalent per pool.** Largest plausible single swap ≈ fund size × position cap ≈ 30k × 30% ≈ 9k; 100k depth keeps price impact small (≥10× rule). Free with mock USDC; we mint + hold the LP position ourselves.
- **#7 — Cut the withdraw queue.** Allow direct `redeem` only while cycle state == IDLE (between cycles) — gives epoch-locking with a fraction of the code. The demo never shows a withdrawal. Don't gold-plate.
- **#8 + #12 — Cycle cadence: 5-minute cycles** (90s voting window, ~3.5 min hold), as a keeper config value, tuned on-site. These two were the same question — the stage demo *is* the live app, so there's no separate "stage timing." Always-on keeper posts the next week's historical prices each cycle; "a week passes" = one cycle tick.
- **#11 — Gas for fresh wallets: server-side drip (primary).** The need isn't "sponsorship" — it's getting free testnet ETH into zero-balance wallets so transactions can be sent at all. **Plan: a backend drip** (~20 lines: one funded wallet sends ~0.001 Base Sepolia ETH to each new wallet on signup) — keeps every wallet a plain address, depends on nobody. Dynamic alternatives (verified 6/12): native sponsorship covers Base Sepolia but is **contact-us gated** + V3-MPC-only; ZeroDev sponsorship is self-serve but wraps wallets in smart accounts (changes addresses → ripples into `verified`/nullifier mappings — avoid). Ask Dynamic booth about native enablement as a nice-to-have; ship the drip.
- **#13 — Hosting: Vercel (web) + Railway (keeper + agents).** Web auto-deploys from GitHub. Keeper must run continuously (Vercel can't) — Railway supports Bun via Dockerfile/nixpacks, and we have prior Railway experience. Use deployed CRE instead if access lands (#5). Be honest in the submission about what runs where.
- **Vault/Governance kept SEPARATE — merge considered + rejected (6/13).** Folding them into one `Fund` contract saves only ~60–100 lines (internal calls don't need an interface anyway — concrete contracts import each other) but risks the **24KB contract-size limit** (ERC-4626 + voting + accuracy + selection + settle + swap calls in one) and makes isolated testing harder under time pressure. The 3-contract design already did the sensible merges (Portfolio+Fees→Vault, Reputation→Governance); money-vs-rules is the natural seam. **Don't merge across it.** Real simplification wins instead: shared off-chain logic (done), `@wagmi/cli` codegen for frontend hooks, one `MockStock` template per ticker, single-source constants in `@chf/shared`.
